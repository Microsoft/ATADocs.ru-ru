# Часто задаваемые технические вопросы о решении Microsoft Advanced Threat Analytics
В этой статье содержатся ответы на часто задаваемые вопросы о решении Microsoft Advanced Threat Analytics (ATA).

## Часто задаваемые вопросы об ATA

|Вопрос|Ответ|
|------------|-----------------------------------|
|Что делать, если шлюз ATA не запускается?|Просмотрите самую последнюю ошибку в текущем журнале ошибок (в месте установки ATA в папке Logs).|
|Как протестировать ATA?|Можно имитировать подозрительные действия, что представляет собой сквозной тест, одним из следующих методов:<br /><br />1.  Проверка DNS с использованием средства Nslookup.exe.<br />2.  Удаленное выполнение с использованием средства psexec.exe.<br /><br />Это нужно выполнить через отслеживаемый контроллер домена, а не через шлюз ATA.|
|Как проверить пересылку событий Windows?|В командной строке каталога **\Program Files\Microsoft Advanced Threat Analytics\Center\MongoDB\bin** можно выполнить следующие действие:<br /><br />mongo ATA --eval "printjson(db.getCollectionNames())" &#124; find /C "NtlmEvents"`|
|Поддерживает ли ATA зашифрованный трафик?|Зашифрованный трафик не анализируется (например, LDAPS, IPSEC ESP).|
|Поддерживает ли ATA защиту Kerberos?|ATA поддерживает механизм защиты Kerberos (также известный как безопасное туннелирование для гибкой аутентификации (FAST)). Однако в случае его активации ATA не сможет обнаруживать атаки Over-pass-the-hash.|
|Как узнать число требуемых шлюзов ATA?|Чтобы определить число требуемых шлюзов, нужно учитывать следующее:<br /><br />— Минимальное количество шлюзов ATA, необходимых для обработки среды Active Directory с точки зрения производительности, зависит от общего объема трафика, который создают контроллеры домена.<br />    Дополнительные сведения о том, как определить, какой объем трафика создается контроллером домена см. в статье [Планирование производительности ATA](/advanced-threat-analytics/PlanDesign/ata-capacity-planning).<br />— Количество шлюзов ATA, которое необходимо для поддержки всех контроллеров домена, (для коммутатора, центра обработки данных и региона) зависит от ограничений зеркального отображения портов. Каждая среда имеет свои собственные требования.|
|Какой объем хранилища требуется для ATA?|На каждый полный день при среднем показателе 1000 пакетов/с требуется 1,5 ГБ свободного места в хранилище.<br /><br />Дополнительные сведения см. в статье [Планирование производительности ATA](/advanced-threat-analytics/PlanDesign/ata-capacity-planning).|
|Почему некоторые учетные записи конфиденциальные?|Такие учетные записи могут быть у участников определенных групп, которые считаются конфиденциальными (например, у участников группы администраторов домена).<br />Чтобы понять, почему учетная запись конфиденциальная, нужно просмотреть, к каким конфиденциальным группам она относится. Группы, к которым она принадлежит, также могут быть конфиденциальными из-за другой группы, поэтому необходимо обнаружить группу с самым высоким уровнем конфиденциальности.|
|Как обеспечить отслеживание виртуального контроллера домена?|Как описано в статье [Настройка зеркального отображения портов](/advanced-threat-analytics/PlanDesign/configure-port-mirroring), шлюз ATA может быть виртуальным или физическим.  <br />Проще всего настроить виртуальный шлюз ATA на каждом узле, где есть виртуальный контроллер домена.<br />Если виртуальные контроллеры домена перемещаются между узлами, необходимо выполнить одно из следующих действий:<br /><br />— Если виртуальный контроллер домена перемещается на другой узел, выполните предварительную настройку шлюза ATA на этом узле, чтобы получать трафик из недавно перемещенного виртуального контроллера домена.<br />— Сопоставьте виртуальный шлюз ATA с виртуальным контроллером домена. Таким образом, они останутся связанными.<br />— Можно использовать виртуальные коммутаторы, которые могут отправлять трафик между узлами.|
|Как осуществить резервное копирование данных ATA?|Есть 2 типа данных, для которых выполняется резервное копирование:<br /><br />— данные о конфигурации ATA, которые хранятся в базе данных и автоматически архивируются каждый час; <br />— сохраняемые ATA сведения о трафике и событиях, резервную копию которых можно создать с помощью любой поддерживаемой процедуры резервного копирования баз данных. Дополнительные сведения см. в статье [Управление базой данных ATA](/advanced-threat-analytics/DeployUse/ata-database-management).|
|Какие атаки и угрозы может обнаружить ATA?|Полный список атак и угроз, которые может обнаружить ATA, см. в статье [Что представляет собой решение Microsoft Advanced Threat Analytics?](/advanced-threat-analytics/Understand/what-is-ata).|
|Какой тип хранилища необходим для ATA?|Мы советуем использовать быстродействующее устройство хранения (диски со скоростью до 7200 об/мин) с малой задержкой (менее 10 мс) и конфигурацией RAID, которая поддерживает большие нагрузки операций записи (кроме RAID 5 и 6, а также производных конфигураций).|
|Какое количество сетевых адаптеров требуется для шлюза ATA?|Для шлюза ATA необходимо как минимум два сетевых адаптера:<br>1. сетевой адаптер для подключения к внутренней сети и центру ATA;<br>2. сетевой адаптер для записи сетевого трафика контроллеров домена путем зеркального отображения порта.<br>* Это не относится к шлюзам упрощенного типа.|
|Какой ATA интегрируется с системами SIEM?|ATA поддерживает двунаправленную интеграцию с системами SIEM:<br>
1. В ATA можно настроить отправку оповещений системного журнала при обнаружении подозрительной активности на любой сервер системы SIEM, который поддерживает формат CEF.<br>2. В ATA можно настроить получение сообщений системного журнала о каждом событии Windows с идентификатором 4776 из [этих систем SIEM](/advanced-threat-analytics/PlanDesign/configure-event-collection#SIEMsupport).|
