# <a name="introduction"></a>Введение

ATA обнаруживает события на следующих этапах продвинутой атаки: разведка, компрометация учетных данных, боковое смещение, повышение привилегий, полное управление доменом и т. д.

Этапы процесса атаки, на которых ATA в настоящее время выявляет угрозы, приведены на рисунке ниже.

![Особое внимание ATA уделяет действиям бокового смещения в процессе атаки](media/attack-kill-chain-small.jpg)

В этой статье приводятся подробные сведения о каждом подозрительном действии на каждом этапе.


## <a name="reconnaissance-using-account-enumeration"></a>Разведывательная атака с использованием перечисления учетных записей.

> [!div class="mx-tableFixed"]
| Описание|Исследование|Рекомендация|Статус|
|------|----|------|----------|
| Атака путем перечисления учетных записей — это прием, с помощью которого злоумышленники пытаются выяснить, существует ли пользователь в сети, подбирая имена учетных записей посредством попыток проверки подлинности Kerberos. Угаданные учетные записи могут использоваться на последующих этапах атаки. | Изучите соответствующий компьютер и постарайтесь определить, есть ли обоснованная причина для запуска большого количества процессов проверки подлинности Kerberos. Подозрение должно вызывать большое количество попыток входа в разные учетные записи, которые завершились неудачно из-за того, что пользователь не существует (ошибка Client_Principal_Unknown), и наличие по крайней мере одной успешной попытки доступа. <br></br>**Исключения**. Обнаружение этой атаки основано на определении многократных попыток входа в несуществующие учетные записи с одного компьютера. Если пользователь допускает ошибку при вводе имени пользователя или домена вручную, попытка проверки подлинности представляется как попытка входа в несуществующую учетную запись. На серверах терминалов, предполагающих вход множества пользователей, большое количество неудавшихся ошибок входа может быть объяснимым. |Изучите процессы, являющиеся причиной создания этих запросов.  Сведения об определении процессов в соответствии с исходным портом см. в записи блога [Как узнать, какой процесс Windows отправляет определенный пакет в сеть?](https://blogs.technet.microsoft.com/nettracer/2010/08/02/have-you-ever-wanted-to-see-which-windows-process-sends-a-certain-packet-out-to-network/)|Средняя|

## <a name="reconnaissance-using-directory-services-enumeration-sam-r"></a>Разведывательная атака с использованием перечисления служб каталогов (SAM-R)

> [!div class="mx-tableFixed"]
| Описание|Исследование|Рекомендация|Статус|
|------|----|------|----------|
| Перечисление служб каталогов — это прием, используемый злоумышленниками для сопоставления структуры каталогов и определения привилегированных учетных записей для дальнейших этапов атаки. Одним из методов, используемых для запроса каталога, является протокол SAM-R. | Установите, почему на рассматриваемом компьютере используется протокол MS-SAMR. Скорее всего, он будет использоваться необычным способом, например для запроса конфиденциальных данных. <br></br>**Исключения**. Обнаружение таких атак основано на профилировании нормального поведения пользователей, выполняющих запросы SAM-R, и оповещении в случае выявления необычных запросов. Когда привилегированные пользователи выполняют вход в компьютеры, которые им не принадлежат, они могут создавать запросы SAM-R, которые определяются как необычные, даже если они создаются в рамках стандартного рабочего процесса. Часто такое происходит в случае с сотрудниками ИТ-отдела. Эти запросы, возникающие в процессе обычной работы, могут помечаться как подозрительные по той причине, что такое поведение ранее не наблюдалось ATA. | В этом случае рекомендуется увеличить период обучения и расширить область действия ATA в домене леса Active Directory.<br></br>[Скачайте и запустите средство SAMRi10](https://gallery.technet.microsoft.com/SAMRi10-Hardening-Remote-48d94b5b). Средство SAMRi10, выпущенное группой разработчиков ATA, усиливает контроль запросов SAM-R в среде. | Средняя|

## <a name="reconnaissance-using-dns"></a>Разведывательная атака с использованием DNS.


> [!div class="mx-tableFixed"]
| Описание|Исследование|Рекомендация|Статус|
|------|----|------|----------|
| На DNS-сервере содержатся сведения о всех компьютерах и службах в вашей сети и их сопоставлении с IP-адресами. Эта информация используется злоумышленниками для определения структуры сети и выявления подходящих компьютеров для проведения последующих этапов атаки. | Установите, почему рассматриваемый компьютер выполняет запрос на полную передачу зоны (AXFR) для получения всех записей в домене DNS. <br></br>**Исключения**. При обнаружении таких атак определяются компьютеры, не являющиеся DNS-серверами, которые отправляют запросы на передачу зоны DNS. Известен ряд сканеров безопасности, которые отправляют подобные запросы на DNS-серверы. <br></br>Кроме того, чтобы избежать ложных срабатываний, убедитесь в том, что ATA может передавать данные из шлюзов ATA на DNS-серверы через порт 53.| Ограничьте возможности передачи зоны, тщательно выбрав узлы, которые могут ее запрашивать. Дополнительные сведения см. в статьях [Обеспечение безопасности DNS](https://technet.microsoft.com/library/cc770474(v=ws.11).aspx) и [Контрольный список: обеспечение безопасности DNS-сервера](https://technet.microsoft.com/library/cc770432(v=ws.11).aspx). |Средняя|

Получив эти сведения, злоумышленник может использовать их для взлома определенных учетных записей и перемещения по сети. | Установите, почему рассматриваемый компьютер выполняет перечисление сеансов SMB.<br></br>**Исключения**. Определение таких атак основано на предположении, что перечисление сеансов SMB не может иметь обоснованного применения в корпоративной сети, но некоторые сканеры безопасности (например, Websense) отправляют такие запросы. | [Используйте средство net cease для усиления контроля над средой](https://gallery.technet.microsoft.com/Net-Cease-Blocking-Net-1e8dcb5b). | Средняя   |

## <a name="brute-force-ldap-kerberos-ntlm"></a>Атака методом подбора (LDAP, Kerberos, NTLM)


> [!div class="mx-tableFixed"]
| Описание|Исследование|Рекомендация|Статус|
|------|----|------|----------|
| При атаки методом подбора злоумышленник пробует угадать пароль, многократно вводя разные пароли. Злоумышленник последовательно проверяет все возможные пароли (или большой набор возможных паролей), пока не будет найден правильный. Подобрав пароль, злоумышленник может войти в сеть как обычный пользователь. В настоящее время ATA поддерживает обнаружение атак методом подбора по горизонтали (несколько учетных записей) с использованием протокола Kerberos или NTLM, а также атак методом подбора по горизонтали и вертикали (одна учетная запись, многократные попытки ввода пароля) с использованием простой привязки LDAP. | Установите, почему на рассматриваемом компьютере происходят неудачные попытки входа в несколько учетных записей (причем число таких попыток для разных учетных записей примерно одинаково) или почему происходит множество неудачных попыток проверки подлинности для одного пользователя. <br></br>**Исключения**. Обнаружение таких атак основано на профилировании обычного поведения учетных записей, проходящих проверку подлинности для доступа к различным ресурсам. Оповещение создается при выявлении необычного поведения. Такое поведение часто наблюдается в скриптах, которые выполняют проверку подлинности автоматически, но используют устаревшие учетные данные (то есть неправильные пароли или имена пользователей). | Длинные сложные пароли обеспечивают первый уровень защиты от атак методом подбора. | Средняя   |

## <a name="sensitive-account-exposed-in-plain-text-authentication-and-service-exposing-accounts-in-plain-text-authentication"></a>Представление конфиденциальной учетной записи при проверке подлинности в виде обычного текста и представление учетной записи службой при проверке подлинности в виде обычного текста


> [!div class="mx-tableFixed"]
| Описание|Исследование|Рекомендация|Статус|
|------|----|------|----------|
| Некоторые службы на компьютере отправляют учетные данные в виде обычного текста даже для привилегированных учетных записей. Злоумышленники, отслеживающие трафик, могут перехватить эти учетные данные и использовать их для несанкционированного доступа. Оповещение инициируется любым паролем к привилегированной учетной записи, передаваемым в виде обычного текста. | Определите компьютер, вызвавший нарушение, и узнайте, почему на нем используются простые привязки LDAP. | Проверьте конфигурацию исходных компьютеров и убедитесь в том, что на них не используется простая привязка LDAP. Вместо применения простой привязки LDAP используйте протокол LDAP SALS или LDAPS. Организуйте многослойную структуру безопасности и ограничьте доступ между уровнями, чтобы предотвратить повышение привилегий. | Низкий — при предоставлении учетных данных службой; средний — для привилегированных учетных записей |

## <a name="honey-token-account-suspicious-activities"></a>Подозрительные действия с использованием учетной записи honeytoken.


> [!div class="mx-tableFixed"]
| Описание|Исследование|Рекомендация|Статус|
|------|----|------|----------|
| Учетные записи honeytoken представляют собой фиктивные учетные записи-ловушки, предназначенные для определения и отслеживания вредоносных действий в сети, которые выполняются с их использованием. Такие учетные записи обычно находятся в состоянии бездействия. Если внезапно регистрируются какие-либо действия, связанные с учетной записью honeytoken, это может свидетельствовать об активности злоумышленника. | Выясните, почему кто-то пытается пройти проверку подлинности с помощью учетной записи honeytoken с данного компьютера. | Просмотрите страницы профилей ATA для других привилегированных учетных записей в среде, чтобы проверить, нет ли других потенциально подозрительных действий. | Средняя   |

Такой прием характерен для атак типа over-pass-the-hash и атак методом подбора. | Установите, почему рассматриваемый компьютер использует протокол проверки подлинности или SMB необычным образом. <br></br>**Исключения**. Такие атаки могут иногда обнаруживаться при использовании допустимых средств для нестандартной реализации протоколов. Это характерно для некоторых приложений, предназначенных для проведения тестов на проникновение. | Проведите мониторинг сетевого трафика и определите, какой процесс создает трафик с необычной реализацией протокола.<br></br>Чтобы определить, имеет ли место атака WannaCry, выполните указанные ниже действия.<br></br> 1.   Скачайте файл Excel с экспортированными данными по подозрительной активности.<br></br>
2.  Откройте вкладку сетевой активности и перейдите к полю "Json", чтобы скопировать соответствующие значения для Smb1SessionSetup и Ntlm.<br></br>
3.  Если Smb1SessionSetup.OperatingSystem имеет значение "Windows 2000 2195", Smb1SessionSetup.IsEmbeddedNtlm — значение "true", а Ntlm.SourceAccountId — значение "NULL", то это атака WannaCry.
| Средний |

## <a name="suspicion-of-identity-theft-based-on-abnormal-behavior"></a>Подозрение на кражу идентификационных данных на основе аномального поведения


> [!div class="mx-tableFixed"]
| Описание|Исследование|Рекомендация|Статус|
|------|----|------|----------|
| После построения поведенческой модели (она создается в течение трех недель на основе не менее чем 50 активных учетных записей) оповещение будет инициироваться при обнаружении любого аномального поведения. Поведение, которое не соответствует модели, созданной для определенной учетной записи пользователя, может указывать на кражу идентификационных данных. | Выясните, почему рассматриваемый пользователь может вести себя необычно. <br></br>**Исключения**. Если ATA имеет частичную область действия (трафик не со всех контроллеров домена маршрутизируется в шлюз ATA), то модель обучения для определенного пользователя будет неполной. Если по истечении трех недель область действия ATA начинает распространяться на весь трафик, то активность пользователя может привести к инициации оповещения. | Разверните ATA на всех контроллерах домена. <br></br>1.  Проверьте, не изменилась ли должность пользователя в организации.<br></br>2.  Проверьте, не является ли пользователь сезонным работником.<br></br>3.  Проверьте, не вернулся ли пользователь к работе после длительного отсутствия.| Средний — для всех пользователей; высокий — для привилегированных пользователей |


## <a name="pass-the-ticket"></a>Атака Pass-the-Ticket.


> [!div class="mx-tableFixed"]
| Описание|Исследование|Рекомендация|Статус|
|------|----|------|----------|
| Атака Pass-the-Ticket — это способ перемещения по сети, при котором злоумышленник похищает билет Kerberos с одного компьютера и использует его для доступа к другому компьютеру путем олицетворения сущности в сети. | Обнаружение таких атак основано на использовании одних и тех же билетов Kerberos на двух компьютерах или нескольких. В некоторых случаях, если ваши IP-адреса часто меняются, ATA не может определить, используются ли разные IP-адреса одним и тем же или разными компьютерами. Эта проблема часто возникает в случае пулов DHCP (VPN, WiFi и т. д.) недостаточного размера и общих IP-адресов (устройства NAT). | Организуйте многослойную структуру безопасности и ограничьте доступ между уровнями, чтобы предотвратить повышение привилегий. | Высокий     |

## <a name="pass-the-hash"></a>Атака Pass-the-Hash.


> [!div class="mx-tableFixed"]
| Описание|Исследование|Рекомендация|Статус|
|------|----|------|----------|
| Во время атаки Pass-the-Hash злоумышленник проходит проверку подлинности на удаленном сервере или в удаленной службе с помощью хэша NTLM пароля пользователя вместо предоставления соответствующего пароля в виде обычного текста, как происходит обычно. | Проверьте, не выполняла ли учетная запись какие-либо аномальные действия в период времени, когда было создано оповещение. | Примените рекомендации, приведенные на странице [Pass-the-Hash](http://aka.ms/PtH). Организуйте многослойную структуру безопасности и ограничьте доступ между уровнями, чтобы предотвратить повышение привилегий. | Высокий|

## <a name="over-pass-the-hash"></a>Атака Over-pass-the-hash.


> [!div class="mx-tableFixed"]
| Описание|Исследование|Рекомендация|Статус|
|------|----|------|----------|
| При проведении атаки Over-Pass-the-Hash используется уязвимость в реализации протокола проверки подлинности Kerberos, заключающаяся в том, что для создания билета Kerberos используется хэш NTLM, что позволяет злоумышленнику проходить проверку подлинности в службах в сети, не зная пароля пользователя. | Переход на более слабое шифрование: установите, почему рассматриваемая учетная запись использует алгоритм RC4 в Kerberos после того, как она получила возможность использовать алгоритм AES. <br></br>**Исключения**. Обнаружение этих атак основано на профилировании методов шифрования, используемых в домене. Оповещение инициируется, если выявлено использование аномального или более слабого метода. В некоторых случаях ATA может определять более слабый метод шифрования как аномальный несмотря на то, что он используется в рамках стандартного рабочего процесса (хотя такие случаи редки). Это может происходить, если такое поведение ранее не наблюдалось ATA. Помочь решить эту проблему может расширение области действия ATA в домене. | Примените рекомендации, приведенные на странице [Pass-the-Hash](http://aka.ms/PtH). Организуйте многослойную структуру безопасности и ограничьте доступ между уровнями, чтобы предотвратить повышение привилегий. | Высокий     |

Дополнительные сведения см. в статьях, посвященных ["серебряному" сертификату атрибута привилегий](https://technet.microsoft.com/library/security/ms11-013.aspx) и [подделке сертификата атрибута привилегий](https://technet.microsoft.com/library/security/ms14-068.aspx). | Высокий     |

## <a name="remote-execution"></a>Удаленное выполнение.


> [!div class="mx-tableFixed"]
| Описание|Исследование|Рекомендация|Статус|
|------|----|------|----------|
| Злоумышленники, получившие учетные данные администратора, могут выполнять удаленные команды на контроллере домена. С их помощью они могут получать постоянный доступ, собирать данные, проводить атаки типа "отказ в обслуживании" (DOS) и выполнять другие действия. | Установите, разрешено ли рассматриваемой учетной записи выполнять такие удаленные команды на контроллере домена. <br></br>**Исключения**. Внедрите механизм [рабочих станций привилегированного доступа](https://technet.microsoft.com/en-us/windows-server-docs/security/securing-privileged-access/securing-privileged-access), чтобы разрешить подключение к контроллерам домена с правами администратора только с защищенных компьютеров. | Низкая      |

|------|----|------|----------|
